GameLink 기술 파트 발표 준비 자료
================================

발표 범위: PPT 7~15번 슬라이드
권장 발표 흐름:
프론트엔드 구조 → 개발·화면 전환 환경 → PWA → 백엔드 API
→ 데이터 검증·데이터베이스 → 인증 보안 → 외부 게임 API


1. 발표 전 반드시 이해할 내용
================================

[React와 TypeScript]

React
- 사용자가 보는 화면을 만드는 라이브러리
- 화면을 로그인, 매칭, 마이페이지 같은 컴포넌트로 분리
- 데이터나 상태가 바뀌면 그에 맞게 화면 갱신
- Layout, BottomNav 같은 공통 UI 재사용

TypeScript
- JavaScript에 데이터 타입 검사를 추가한 언어
- 사용자 ID는 숫자, 닉네임은 문자열처럼 데이터 형태를 명시
- 프론트와 백엔드 사이의 데이터 실수를 개발 단계에서 발견
- 실제 서버 응답을 자동 검증하는 것은 아니며, 개발자가 정의한 타입을 기준으로 코드를 검사


[Vite와 React Router]

Vite
- 개발 중 React 앱을 실행하는 로컬 서버 제공
- 코드 수정 결과를 브라우저에 빠르게 반영
- 개발 환경에서 API 요청을 백엔드로 전달하는 프록시 제공
- 배포할 때 코드를 브라우저용으로 변환하고 압축

React Router
- URL에 따라 보여줄 React 화면을 결정
- 새로운 HTML 파일을 매번 받지 않고 앱 내부에서 화면 교체
- HashRouter는 # 뒤의 주소를 React가 처리하게 해 정적 서버의 새로고침 404 문제 방지


[PWA]

- 웹과 앱의 장점을 결합한 방식
- 앱스토어를 거치지 않고 웹사이트에서 설치 가능
- 홈 화면 아이콘과 독립 실행 화면 제공
- 서비스워커가 CSS, JavaScript, 이미지 같은 정적 파일 캐싱
- 서버가 필요한 로그인·매칭까지 완전히 오프라인으로 작동하는 것은 아님


[FastAPI와 Pydantic]

FastAPI
- 백엔드 API를 만드는 Python 웹 프레임워크
- 인증, 프로필, 매칭 같은 기능을 라우터별로 분리
- 자동 API 문서, 타입 힌트, 비동기 처리 지원

Pydantic
- 백엔드가 받은 요청 데이터가 올바른지 검사
- 필수 정보 누락, 잘못된 타입, 이메일 형식 등을 검증
- TypeScript는 프론트 개발 과정에서 검사
- Pydantic은 서버가 실제 요청을 받을 때 검사


[SQLAlchemy와 트랜잭션]

SQLAlchemy
- Python 코드의 객체와 데이터베이스 테이블을 연결하는 ORM
- 직접 SQL을 반복해서 작성하는 부담을 줄임
- 사용자, 프로필, 대기열, 매치 데이터를 모델로 관리

트랜잭션
- 여러 데이터 작업을 하나의 작업 단위로 처리
- 전부 성공하면 저장하고, 중간에 실패하면 되돌림
- 여러 사용자가 동시에 매칭될 때 중복 매칭과 데이터 불일치를 방지하는 데 중요


[JWT와 bcrypt]

bcrypt
- 비밀번호를 원문으로 저장하지 않고 복구하기 어려운 해시값으로 변환
- 로그인할 때 입력 비밀번호와 저장된 해시를 비교

JWT
- 로그인 성공 후 발급하는 인증 토큰
- 프론트가 API 요청의 Authorization 헤더에 포함
- 서버는 토큰을 검증해 로그인한 사용자를 식별
- JWT가 암호화된 비밀번호라는 설명은 틀림


[외부 게임 API]

- Riot ID나 피파 닉네임으로 외부 게임 데이터를 조회
- 조회한 티어·전적을 사용자 프로필이나 매칭 조건에 반영
- API 키는 프론트에 넣지 않고 백엔드 환경변수로 관리
- 호출 횟수 제한에 대비해 캐싱이나 재호출 제한 필요


2. 5분 발표 대본
================================

[0:00~0:20 기술 구조 소개]

지금부터 GameLink의 프론트엔드와 백엔드 기술 구조를 설명하겠습니다.
사용자가 화면에서 게임과 조건을 선택하면 프론트엔드가 API를 통해
백엔드에 요청하고, 백엔드는 데이터를 검증한 뒤 데이터베이스와
외부 게임 API를 이용해 결과를 반환하는 구조입니다.


[0:20~1:10 React와 TypeScript]

먼저 프론트엔드는 React와 TypeScript를 사용했습니다.
React는 하나의 큰 화면을 작은 부품인 컴포넌트로 나눠 만드는 도구입니다.
저희는 로그인, 매칭, 마이페이지를 각각 페이지 컴포넌트로 분리했고,
여러 페이지에서 반복되는 Layout과 BottomNav는 공통 컴포넌트로 재사용했습니다.

또한 사용자가 게임을 선택하거나 서버에서 매칭 결과가 도착해 상태가 바뀌면,
React가 그 상태에 맞게 화면을 갱신합니다.

TypeScript는 사용자 ID는 숫자, 닉네임은 문자열처럼 데이터 종류를
미리 정의하는 언어입니다. 이를 통해 프론트와 백엔드가 주고받는 데이터에서
필수 정보가 빠지거나 숫자와 문자열을 혼동하는 문제를 개발 단계에서
발견할 수 있었습니다.


[1:10~2:00 Vite와 React Router]

개발 및 빌드 도구는 Vite를 사용했습니다.
Vite는 개발 중 로컬 서버를 제공하고, 코드를 수정하면 변경 사항을
브라우저에 빠르게 반영합니다.

개발 환경에서는 프론트와 백엔드 주소가 다르기 때문에 API 프록시도 사용했습니다.
프론트가 Vite 서버의 /api 주소로 요청하면, Vite가 그 요청을 실제 백엔드로
대신 전달하는 방식입니다.

페이지 이동은 React Router로 구현했습니다.
일반 웹페이지처럼 이동할 때마다 새로운 HTML을 받는 것이 아니라,
URL에 맞는 React 컴포넌트로 화면을 교체합니다.

또한 HashRouter를 사용해 # 뒤의 주소를 React가 처리하도록 했고,
정적 배포 환경에서 새로고침했을 때 서버가 존재하지 않는 파일을 찾다가
404가 발생하는 문제를 줄였습니다.


[2:00~2:30 PWA]

이 프로젝트는 PWA 방식도 적용했습니다.
PWA는 웹사이트의 접근성과 모바일 앱의 편리함을 결합한 방식입니다.
사용자는 앱스토어를 거치지 않고 웹사이트를 홈 화면에 설치해
독립된 앱처럼 실행할 수 있습니다.

서비스워커는 JavaScript, CSS와 이미지 같은 정적 파일을 캐시에 저장해
재접속 속도를 개선합니다. 다만 로그인과 실시간 매칭은 백엔드 서버가
필요하기 때문에 모든 기능이 오프라인에서 작동하는 것은 아닙니다.


[2:30~3:15 FastAPI와 Pydantic]

백엔드는 FastAPI를 사용했습니다.
인증, 프로필, 매칭 등의 API를 기능별 라우터로 분리해 각 기능을
독립적으로 관리했습니다. 이러한 구조는 코드 유지보수와 팀 협업에 유리하고,
FastAPI가 자동으로 API 문서를 만들어 준다는 장점도 있습니다.

클라이언트에서 요청이 들어오면 먼저 인증과 데이터 검증을 거친 뒤
실제 기능을 수행합니다. 데이터 검증에는 Pydantic을 사용했습니다.
예를 들어 이메일 형식이 잘못됐거나 필수 정보가 빠졌거나 예상한 데이터
타입과 다르면, 비즈니스 로직을 실행하기 전에 요청을 거절합니다.

TypeScript가 프론트 개발 과정에서 실수를 발견한다면,
Pydantic은 서버에 실제로 들어온 데이터를 검사한다는 차이가 있습니다.


[3:15~4:00 데이터베이스]

데이터베이스는 SQLAlchemy를 이용해 모델로 관리했습니다.
사용자 기본 정보, 게임 프로필, 매칭 대기열과 매치 결과를 각각
관계형 테이블로 분리했습니다. 이를 통해 중복 데이터를 줄이고
한 사용자의 프로필과 매치 기록을 관계로 연결할 수 있습니다.

특히 매칭에서는 여러 사용자가 동시에 접근할 수 있기 때문에
트랜잭션이 중요합니다. 트랜잭션은 관련된 데이터 작업을 하나의 묶음으로
처리해 전부 성공하면 저장하고, 중간에 실패하면 원래 상태로 되돌리는 방식입니다.
이를 통해 한 사용자가 여러 매치에 동시에 포함되는 것과 같은
데이터 불일치를 방지할 수 있습니다.


[4:00~4:35 인증 보안]

인증에는 bcrypt와 JWT를 사용했습니다.
회원가입 시 비밀번호를 그대로 저장하지 않고 bcrypt로 해시 처리해 저장합니다.
로그인할 때 입력된 비밀번호와 저장된 해시를 비교하고,
일치하면 JWT 액세스 토큰을 발급합니다.

이후 프론트는 API 요청마다 Authorization 헤더에 토큰을 포함합니다.
백엔드는 토큰의 유효성을 확인한 다음 로그인 사용자에게만
필요한 기능을 실행합니다.


[4:35~5:00 외부 API와 마무리]

마지막으로 Riot과 피파 같은 외부 게임 API를 이용해 사용자의 티어와
전적 정보를 동기화하는 구조를 설계했습니다.
외부 API 키는 노출될 수 있는 프론트가 아니라 백엔드 환경변수로 관리하고,
호출 제한을 고려해 중복 호출을 줄이는 방식이 필요합니다.

정리하면 React와 TypeScript로 사용자 화면과 데이터 안정성을 확보하고,
FastAPI와 Pydantic으로 API와 데이터 검증을 구성했습니다.
SQLAlchemy로 데이터를 관리하고, JWT와 bcrypt로 인증 보안을 구현해
전체 사용자 흐름을 연결했습니다.


3. 발표 전에 실제 구현 여부를 확인할 부분
================================

다음 기능이 백엔드에 실제로 구현되지 않았다면 “구현했다”고 단정하지 말고
“설계했다” 또는 “추후 적용할 수 있다”고 표현해야 합니다.

- 전남대학교 이메일 인증
- 랭킹 API
- 피파 외부 API 연동
- 외부 API 캐싱과 호출 횟수 추적
- 트랜잭션을 이용한 중복 매칭 방지
- 이메일 인증 사용자만 매칭·랭킹 사용
- Riot 전적 및 최근 게임 기록 저장

현재 확인한 프론트엔드에서는 Riot ID 동기화 API는 연결되어 있지만,
피파 계정은 화면 중심으로 구현돼 있습니다.

피파 API가 실제로 구현되지 않았다면 다음과 같이 설명합니다.

“Riot 계정은 실제 API 연동을 적용했고, 피파를 포함한 나머지 게임은
향후 같은 구조로 확장할 수 있도록 UI와 데이터 흐름을 설계했습니다.”


4. 예상 질문과 답변
================================

Q. React와 FastAPI는 어떤 관계인가요?
A. React는 사용자 화면을 담당하고, FastAPI는 로그인, 매칭, 데이터 저장 같은
   서버 기능을 API로 제공합니다.

Q. TypeScript와 Pydantic의 차이는 무엇인가요?
A. TypeScript는 프론트 코드를 작성할 때 타입을 검사하고,
   Pydantic은 서버에 실제로 도착한 요청 데이터를 실행 중에 검증합니다.

Q. 왜 폴링을 사용했나요?
A. 매칭 완료 시점을 확인하기 위해 프론트에서 일정 간격으로 상태 API를
   조회했습니다. 규모가 커지면 WebSocket으로 개선할 수 있습니다.

Q. JWT에 비밀번호가 들어가나요?
A. 아니요. 비밀번호는 bcrypt로 해시 저장하고, JWT에는 사용자를 식별하고
   토큰을 검증하는 데 필요한 정보만 담습니다.

Q. PWA면 오프라인 매칭도 가능한가요?
A. 정적 화면 파일은 캐시할 수 있지만 로그인과 매칭은 서버 데이터가
   필요하므로 인터넷 연결이 필요합니다.

Q. API 프록시는 왜 사용했나요?
A. 개발 환경에서 프론트와 백엔드 주소가 다르기 때문에 요청 경로를
   단순화하고 교차 출처 문제를 줄이기 위해 사용했습니다.

Q. HashRouter는 왜 사용했나요?
A. # 뒤의 화면 주소를 서버가 아닌 React가 처리하게 하여,
   정적 배포 환경에서 새로고침할 때 발생할 수 있는 404를 방지하기 위해
   사용했습니다.


5. 마지막 핵심 요약
================================

- React: 컴포넌트와 상태를 이용한 사용자 화면 구현
- TypeScript: 프론트 코드와 API 데이터의 타입 오류 방지
- Vite: 개발 서버, API 프록시, 배포용 빌드
- React Router: URL에 따른 화면 이동
- PWA: 웹사이트를 설치 가능한 앱처럼 제공
- FastAPI: 인증, 프로필, 매칭 등의 백엔드 API 제공
- Pydantic: 서버 요청과 응답 데이터 검증
- SQLAlchemy: 데이터베이스 테이블과 관계 관리
- bcrypt: 비밀번호 해시 저장
- JWT: 로그인 사용자를 확인하는 인증 토큰
- 외부 게임 API: 사용자의 티어와 전적 정보 동기화
